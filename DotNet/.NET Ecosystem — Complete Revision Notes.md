# .NET Ecosystem — Complete Revision Notes

---

## 1. The Three Platforms: Framework vs Core vs Standard

### .NET Framework
- The **original** Windows-only platform, launched in **2002**
- Deeply integrated with Windows (Win32, WPF, WinForms, WebForms, COM)
- Now in **maintenance mode** — no new features, only security fixes
- Latest version: **4.8.1 (2022)** — considered legacy
- Use only if maintaining old/existing apps

### .NET Core
- Microsoft's **cross-platform rewrite**, launched in **2016**
- Runs on Windows, Linux, macOS
- Faster, modular, open-source, cloud-friendly
- Versions: 1.0 → 1.1 → 2.x → 3.x → then rebranded
- **Superseded by .NET 5+** — migrate if still on Core

### .NET Standard
- NOT a runtime — it's a **specification / API contract**
- Defined a set of APIs that both Framework and Core agreed to implement
- Let you write a library once and have it work on both platforms
- Now **largely obsolete** since .NET 5+ unified everything
- Only use if your library needs to support both old Framework and new .NET

### The Unification: .NET 5 and beyond
- From **.NET 5 (2020)** onwards, Microsoft dropped "Core" and "Framework" branding
- Everything unified under just **".NET"**
- Released annually every November
- .NET 8 = LTS (Long Term Support, until Nov 2026)
- .NET 9 = Latest (released Nov 2024)
- .NET 10 = Next LTS (expected Nov 2025)

### Comparison Table

| Platform        | Status              | Cross-platform    | Use today?              |
|-----------------|---------------------|-------------------|--------------------------|
| .NET Framework  | Legacy/Maintenance  | Windows only      | Only for old apps        |
| .NET Core       | Superseded          | Yes               | Migrate to .NET 8/9      |
| .NET Standard   | Obsolete            | Yes (spec only)   | Only for lib compat      |
| **.NET 8 / 9**  | **Current**         | **Yes**           | **Yes — use this**       |

## 2. WPF & WinForms on Modern .NET

### Key Point
Even though .NET is cross-platform, **WPF and WinForms are still Windows-only**.  
They were ported to .NET Core 3.1+ and modern .NET, but never made cross-platform.

### Why Still Windows-Only?
WPF and WinForms are deeply tied to:
- Win32 APIs
- GDI+ (graphics)
- DirectX
- Windows message loops

Abstracting all of that for Linux/macOS would mean rewriting the entire UI stack.

### How to Use Them Today

The key is the **`-windows` Target Framework Moniker (TFM)**:

**WinForms on .NET 8:**
```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>WinExe</OutputType>
    <TargetFramework>net8.0-windows</TargetFramework>
    <UseWindowsForms>true</UseWindowsForms>
  </PropertyGroup>
</Project>
```

**WPF on .NET 8:**
```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>WinExe</OutputType>
    <TargetFramework>net8.0-windows</TargetFramework>
    <UseWPF>true</UseWPF>
  </PropertyGroup>
</Project>
```

### Modern Windows Desktop Options

| Technology   | Runs on .NET 8/9? | Cross-platform? | Status                        |
|--------------|-------------------|-----------------|-------------------------------|
| WinForms     | Yes               | Windows only    | Active, maintained            |
| WPF          | Yes               | Windows only    | Active, maintained            |
| UWP          | Partial           | Windows only    | Being replaced                |
| WinUI 3      | Yes               | Windows only    | Microsoft's recommended modern UI |

## 3. How .NET Became Cross-Platform

### The Core Idea: IL (Intermediate Language)

When you compile C# code, it does **NOT** produce native machine code directly.  
It produces **IL (Intermediate Language)** — also called MSIL or CIL.

```
Your C# Code
     ↓  (Roslyn Compiler)
  IL Code  ← platform-neutral bytecode (like Java's bytecode)
     ↓  (CLR / CoreCLR Runtime)
Native Machine Code  ← happens at runtime, per OS/CPU
```

### What's Inside the EXE?

```
myapp.exe
├── PE Header          ← tiny Windows wrapper (just enough to launch the runtime)
├── IL Bytecode        ← your actual compiled code (platform neutral)
├── Metadata           ← type info, references
└── Manifest           ← assembly info
```

On Linux, the same IL is packaged as a `.dll` launched via:
```bash
dotnet myapp.dll   # Linux/macOS
myapp.exe          # Windows (same IL inside, different wrapper)
```

### The Runtime is the Secret: CoreCLR

Microsoft rewrote **CoreCLR** natively for each platform:

| Platform | Runtime Binary      |
|----------|---------------------|
| Windows  | `coreclr.dll`       |
| Linux    | `libcoreclr.so`     |
| macOS    | `libcoreclr.dylib`  |

Each platform's CoreCLR:
1. Loads the IL bytecode
2. JIT-compiles it to native CPU instructions at runtime
3. Manages memory, GC, threading, etc.

### Three Publishing Modes

**1. Framework Dependent (default — smallest output)**
```bash
dotnet publish -c Release
# Produces small .dll — requires dotnet runtime on target machine
```

**2. Self-Contained (includes runtime)**
```bash
dotnet publish -c Release -r linux-x64 --self-contained
# Bundles entire runtime with app (~60-80MB)
# No dotnet install needed on target machine
```

**3. Single File (everything in one binary)**
```bash
dotnet publish -c Release -r linux-x64 --self-contained -p:PublishSingleFile=true
# One executable file
# On Linux → ELF binary
# On Windows → PE .exe
```

### Java Parallel

| | Java | .NET |
|---|---|---|
| Bytecode | `.class` / `.jar` | IL in `.dll` / `.exe` |
| Runtime | JVM | CLR / CoreCLR |
| JIT Compiler | HotSpot JIT | RyuJIT |
| Cross-platform unit | JAR file | Assembly (DLL) |

## 4. The Complete Compilation Pipeline

### All the Components and When They Run

```
YOUR MACHINE (Build Time)                TARGET MACHINE (Runtime)
─────────────────────────                ────────────────────────
dotnet build                             dotnet myapp.dll
      ↓                                        ↓
  ROSLYN                                 CORECLR boots
  (compile time)                               ↓
      ↓                                  Loads IL + dependencies
  C# → IL                                      ↓
      ↓                                   RYUJIT
  MyApp.dll ── you ship this ──→         (per method, first call)
                                               ↓
                                         IL → Native Machine Code
                                               ↓
                                           App runs! 🚀
```

---

### Stage 1: Writing Code
```csharp
// Program.cs — just a plain text file
Console.WriteLine("Hello, World!");
```

---

### Stage 2: Roslyn Compiler (Build Time, Your Machine)

Triggered by `dotnet build` or `dotnet run`.

```
Program.cs (text)
    ↓
┌─────────────────────────────┐
│         ROSLYN              │
│                             │
│  1. Lexing & Tokenizing     │  "Console", ".", "WriteLine", "(" ...
│  2. Parsing → AST           │  Abstract Syntax Tree
│  3. Semantic Analysis       │  Type checking, scope resolution
│  4. Code Generation         │  Emits IL instructions
└─────────────────────────────┘
    ↓
  MyApp.dll  (contains IL bytecode + Metadata)
```

**Roslyn's job ends here. It only produces IL — never machine code.**

What IL looks like:
```il
IL_0000: ldstr   "Hello World"              // push string onto stack
IL_0005: call    System.Console.WriteLine   // call the method
IL_000a: ret                                // return
```
This IL is **CPU-agnostic** — no mention of x64, ARM, or any architecture.

---

### Stage 3: Ship the DLL

The compiled `.dll` is what you distribute. The **same file** works on:
- Windows x64
- Linux x64
- macOS ARM64 (Apple Silicon)
- ARM (Raspberry Pi)

---

### Stage 4: CoreCLR (Runtime, Target Machine)

When someone runs your app, CoreCLR boots:

```
┌──────────────────────────────────────┐
│              CoreCLR                 │
│                                      │
│  1. Class Loader                     │  Loads DLL + dependencies
│  2. Type System                      │  Builds type metadata in memory
│  3. Memory Manager                   │  Sets up managed heap
│  4. Garbage Collector (GC)           │  Ready to manage memory
│  5. Thread Manager                   │  Sets up threading
│  6. Security & Verification          │  Validates IL is safe
│  7. Hands IL to RyuJIT               │  → next stage
└──────────────────────────────────────┘
```

CoreCLR is OS-native — it bridges your app to OS-specific APIs:
- Windows APIs on Windows
- POSIX/Linux syscalls on Linux
- macOS APIs on macOS

---

### Stage 5: RyuJIT (Runtime, Per Method, First Call)

RyuJIT is the **Just-In-Time compiler** living inside CoreCLR:

```
IL Bytecode (platform neutral)
        ↓
┌─────────────────────────────────────┐
│              RyuJIT                 │
│                                     │
│  1. Takes IL method by method       │
│  2. Optimizes (inlining, loops...)  │
│  3. Generates native CPU opcodes    │
└─────────────────────────────────────┘
        ↓
  x64 machine code    (Intel/AMD)
  ARM64 machine code  (Apple M1/M2, mobile)
  x86 machine code    (32-bit systems)
```

**JIT is LAZY — methods compiled only on first call, then cached in RAM:**
```
Main() called first time   → RyuJIT compiles Main()         → runs
WriteLine() called         → RyuJIT compiles WriteLine()    → runs
WriteLine() called again   → already compiled, skip JIT     → runs directly
```

---

### Same DLL, Different OS

```
SAME MyApp.dll
      │
      ├──→  Windows x64
      │         CoreCLR (coreclr.dll) + RyuJIT → x64 opcodes
      │         Uses Win32 APIs internally
      │
      ├──→  Linux x64
      │         CoreCLR (libcoreclr.so) + RyuJIT → x64 opcodes
      │         Uses POSIX/Linux syscalls internally
      │
      ├──→  macOS ARM64
      │         CoreCLR (libcoreclr.dylib) + RyuJIT → ARM64 opcodes
      │         Uses macOS APIs internally
      │
      └──→  Raspberry Pi (Linux ARM)
                CoreCLR (libcoreclr.so for ARM) + RyuJIT → ARM opcodes
                Uses Linux syscalls internally
```

---

### Component Summary Table

| Component    | Role                          | When                          |
|--------------|-------------------------------|-------------------------------|
| **Roslyn**   | C# → IL compiler              | Build time, your machine      |
| **IL**       | Platform-neutral bytecode     | The artifact you ship         |
| **CoreCLR**  | Runtime host, GC, OS bridge   | When app starts on target     |
| **RyuJIT**   | IL → Native machine code      | First call of each method     |
| **Native Code** | Actual CPU instructions    | After JIT, cached in RAM      |

---

### Bonus: Native AOT (Ahead of Time Compilation)

```bash
dotnet publish -c Release -r linux-x64 -p:PublishAot=true
```

```
Normal JIT:   C# →(Roslyn)→ IL →(RyuJIT at runtime)→ Machine Code
Native AOT:   C# →(Roslyn)→ IL →(RyuJIT at BUILD time)→ Native Binary
```

Benefits of AOT:
- No JIT warmup on target machine
- No CoreCLR needed on target
- Truly native binary per platform
- Faster startup, smaller memory footprint
- Must build separately for each OS target
- Great for microservices and CLI tools

## 5. Docker and Kubernetes

### The Problems They Solve

```
"It works on my machine!"
         ↓
    Docker fixes this

"But now I have 100 instances to manage!"
         ↓
    Kubernetes fixes this
```

---

### Docker — "Package Everything Together"

**The Problem Without Docker:**
```
Developer Machine          Production Linux Server
──────────────────         ───────────────────────
.NET 8 installed     ≠     .NET 6 installed
Ubuntu 22            ≠     CentOS 7
libssl 3.0           ≠     libssl 1.1
env vars set         ≠     env vars missing

Result → App crashes on server 💥
```

**What a Docker Image Contains:**
```
┌─────────────────────────────┐
│       Docker Image          │
│                             │
│  Your App (.dll)            │  ← IL bytecode
│  .NET 8 Runtime             │  ← CoreCLR, RyuJIT
│  Linux Libraries            │  ← libssl, libc, etc.
│  Base OS (Alpine/Ubuntu)    │  ← minimal Linux
└─────────────────────────────┘
```

This image runs **identically** on any machine with Docker installed.

### Dockerfile for a .NET App

```dockerfile
# Stage 1 — Build (Roslyn compiles here)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /app
COPY . .
RUN dotnet publish -c Release -o /out    # Roslyn runs → produces IL

# Stage 2 — Runtime image (much smaller)
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime
WORKDIR /app
COPY --from=build /out .                 # Only copy IL output
ENTRYPOINT ["dotnet", "myapp.dll"]       # CoreCLR + RyuJIT runs here
```

### .NET Docker Image Sizes

| Image                     | Size    | Contains                          |
|---------------------------|---------|-----------------------------------|
| `dotnet/sdk:8.0`          | ~700MB  | Roslyn, build tools, everything   |
| `dotnet/aspnet:8.0`       | ~200MB  | CoreCLR runtime + ASP.NET         |
| `dotnet/runtime:8.0`      | ~190MB  | CoreCLR runtime only              |
| `dotnet/runtime-deps`     | ~120MB  | Just OS deps (use with AOT)       |

### Container vs Virtual Machine

```
VIRTUAL MACHINE                    DOCKER CONTAINER
───────────────────────────        ────────────────────────────
Your App                           Your App
.NET Runtime                       .NET Runtime
Full Guest OS (GBs)                Minimal OS libs only
Hypervisor                         Docker Engine
Host OS                            Host OS (shared kernel)

Heavy, slow to start (~mins)       Lightweight, starts in ms
```

Containers **share the host OS kernel** — they don't boot a full OS.

### Kubernetes — "Manage 100s of Containers"

**The Problem Without K8s:**
- Traffic spike → need 50 containers instead of 2
- One server dies → need to reschedule pods
- New version → need zero-downtime deployment
- Managing all this manually is impossible at scale

**K8s Cluster Structure:**
```
┌─────────────────────────────────────────────────┐
│                  Kubernetes Cluster              │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │           Control Plane                  │   │
│  │  - API Server                            │   │
│  │  - Scheduler                             │   │
│  │  - Controller Manager                    │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  Node 1  │  │  Node 2  │  │  Node 3  │      │
│  │ [Pod]    │  │ [Pod]    │  │ [Pod]    │      │
│  │ [Pod]    │  │ [Pod]    │  │ [Pod]    │      │
│  └──────────┘  └──────────┘  └──────────┘      │
└─────────────────────────────────────────────────┘
```

**What K8s Handles Automatically:**

| Situation            | K8s Response                                      |
|----------------------|---------------------------------------------------|
| Traffic spike        | Auto-scales pods (HorizontalPodAutoscaler)        |
| Pod crashes          | Detects and restarts automatically (self-healing) |
| Node (server) dies   | Reschedules pods to healthy nodes                 |
| New version deploy   | Rolling update with zero downtime                 |
| Load balancing       | Distributes traffic across all pods               |

### The Full Modern .NET Deployment Pipeline

```
Developer writes C# code
         ↓
    git push
         ↓
CI/CD Pipeline (GitHub Actions / Azure DevOps)
    ├── Roslyn compiles C# → IL
    ├── Tests run
    ├── docker build → container image
    └── docker push → Container Registry (Docker Hub / ECR / ACR)
         ↓
    Kubernetes pulls image
         ↓
    Schedules pods across nodes
         ↓
    Each pod:
        CoreCLR boots
        RyuJIT compiles IL → native code
        App serves traffic 🚀
         ↓
    Traffic spike? → K8s auto-scales
    Pod dies?      → K8s auto-restarts
    New version?   → K8s rolling update
```

```
┌────────────────────────────────────┐

└────────────────────────────────────┘
```
